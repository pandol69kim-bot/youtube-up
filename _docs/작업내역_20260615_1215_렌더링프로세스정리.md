## 작업 내용

렌더링 프로세스의 세 가지 문제점을 수정하고 정리.

## 문제점 및 수정 사항

### 1. stale `rendering` 레코드 고착 문제
- **문제**: 서버가 렌더링 중 재시작되면 해당 Video 레코드가 영구적으로 `rendering` 상태로 남음 (video id=3이 이 케이스)
- **수정**: `main.py` 서버 시작(lifespan) 시 `_fix_stale_rendering()` 함수 실행 — `rendering` 상태 레코드를 전부 `failed`로 정정

### 2. 페이지 새로고침 시 기존 영상 상태 초기화
- **문제**: 플레이리스트를 선택하거나 페이지를 새로고침하면 `video` state가 null이 되어 이미 렌더링된 영상 정보를 잃어버림
- **수정**:
  - `GET /video/latest/{playlist_id}` 엔드포인트 추가 (ready/uploaded/scheduled 상태 중 최신 영상 반환)
  - `api.ts`에 `latestVideo()` 함수 추가
  - `refreshTracks()`에서 플레이리스트 트랙 로드 후 자동으로 최신 영상도 로드

### 3. 배경색 하드코딩
- **문제**: `renderVideo()` 호출 시 배경색이 `"#12372a"`로 하드코딩되어 있어 변경 불가
- **수정**: `bgColor` state 추가 + 트랙 편집 헤더에 컬러 피커 UI 추가 (기본값 `#101827`)

## 변경된 파일

- `backend/app/main.py`: `_fix_stale_rendering()` 함수 추가, lifespan에서 호출
- `backend/app/routers/video.py`: `GET /video/latest/{playlist_id}` 엔드포인트 추가, `DONE_STATUSES` 상수 추출
- `frontend/app/api.ts`: `latestVideo(playlistId)` 함수 추가
- `frontend/app/page.tsx`: `bgColor` state 추가, `refreshTracks`에서 기존 영상 자동 로드, 컬러 피커 UI 추가

## 렌더링 플로우 요약 (정리 후)

```
[플레이리스트 선택]
  → refreshTracks(playlistId)
  → GET /tracks?playlist_id=N  →  트랙 목록 로드
  → GET /video/latest/N        →  기존 완료 영상 자동 복원 (없으면 null)

[렌더링 버튼 클릭]
  → POST /video/render  {playlist_id, background_color}
  → Video 생성 (status=rendering)
  → render_playlist_video() — ffmpeg로 MP4 생성
  → status = ready (성공) / failed (실패)
  → chapters 자동으로 설명란에 삽입

[서버 재시작]
  → _fix_stale_rendering()
  → rendering 상태 레코드 → failed 로 일괄 정정
```
